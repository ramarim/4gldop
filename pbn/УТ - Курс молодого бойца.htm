Что можно сделать с помощью универсальных транзакций?

Универсальные транзакции 41D (УТ) - мощный инструмент системы БИСКВИТ, предоставляющий богатый функционал для работы с объектами системы.

ВНИМАНИЕ! Производительность УТ уступает производительности процедур, реализующих тот же функционал, но написанных непосредственно на языке Progress 4GL.


Перед тем как приступить к описанию возможностей УТ хочу обратить особое внимание на то, что в основе механизма УТ лежит метасхема. При обработке шаблонов УТ используют данные из метасхемы для того, чтобы знать, например, какие реквизиты объекта должны быть обязательно заполнены, а какие нет, какую процедуру вызвать для отображения формы заполнения реквизитов объекта и т.п.

Ниже в данной статье я опишу лишь то, что сам делал с помошью УТ. Возможно кто-нибудь другой предложит еще варианты их использования.

  1. Работа с договорами - открытие, изменение реквизитов договоров, создание новых и изменение существующих условий, работа с картотекой счетов.
  2. Работа с документами - создание документов и бухгалтерских проводок, изменение существующих.
  3. Импорт и экспорт данных - обмен информацией с другими системами по средством текстовых файлов (формат обмена определяется алгоритмом, прописанным в УТ).
  4. Групповая обработка объектов - можно задать в УТ условие выборки объектов, и для каждого выбранного объекта выполнить какие-либо действия.
  5. Работа с классификаторами - чтение, создание новых элементов, изменение.

Вообще, с помощью УТ можно создавать и изменять любые объекты, определенные в метасхеме системы.

Переменные и область видимости

Встроенный язык УТ является (в общем смысле) интерпретируемым языком. Как и во многих других языках подобного типа, переменные, используемые в языке, не требуют предварительного описания.

Переменная может хранить данные любого типа: текст, число, дату.

Переменные всегда хранятся в памяти как текст и на системном уровне нет информации о действитльном типе значения переменной. Для использования переменных в арифметических выражениях или выражениях с датами необходимо постоянное приведение с помощью функций DEC, INT, DATE.

Чтобы получить доступ к значению переменной, нужно обратиться к ней по ссылке с помощью оператора @ ("собака") либо функции GETVAR

Все переменные можно разделить на три группы:

  • атрибуты объектов;
  • функциональные переменные;
  • пользовательские переменные.

Атрибуты объектов - это переменные, ссылающиеся на реквизиты объектов определенного в метасхеме класса. Имя переменной-атрибута состоит из двух частей: имени атрибута и ссылки на объект. Примеры переменных-атрибутов показаны ниже. 

  // в УТ определен шаблон №10 создания (поиска, модификации) договора, чтобы получить номер договора
// нужно обратиться к переменной
@cont-code(10);

// в УТ определен шаблон №45 создания (поиска, модификации) документа, чтобы получить содержание операции
// нужно обратиться к переменной 
@details(45)


Имя переменной-атрибута всегда должно совпадать с именем реквизита, определенного в метасхеме класса, на который эта переменная-атрибут ссылается.

Функциональные переменные - переменные, создаваемые в процессе выполнения функций встроенного языка УТ или в процессе обработки шаблонов УТ. Например функция ГР_ТРАНЗАКЦИЯ при каждом своем запуске создает перменную @__counter, содержащую номер текущей итерации.

Пользовательские переменные - переменные, определяемые и используемые в коде УТ. Пример использования пользовательских переменных показан ниже. 

@__var1 = 3;
@__var2 = 7;
@__var3 = INT(@__var1) + @__var2
</code>

Область видимости

Область видимости в УТ вытекает из технических обособенностей обработки УТ:

  • каждый шаблон, определенный в УТ, обрабатывается обособлено;
  • все шаблоны обрабатываются последовательно сверху-вниз;
  • каждая УТ может вызвать любую другую УТ;
  • после обработки вызванной УТ, продолжается обработка текущей УТ.


Область видимости для каждого типа переменных особая. Для переменных-атрибутов область видимости простирается на отдельно взятую УТ. Т.е. из любого шаблона объекта можно сослаться на реквизит объекта, определенного в любом другом шаблоне данной транзакции, в том числе и на реквизит объекта текущего шаблона. Обратиться к реквизиту объекта, определенного в вызывающей транзакции можно с помощью функции GETVAR, одним из необязательных параметров которой является символьный код транзакции.

Для функциональных и пользовательских переменных область видимости имеет структуру, которая показана на рисунке ниже.

На рисунке показаны две транзакции: транзакция УТ-1 в первом шаблоне вызывает транзакцию УТ-2.

Область видимости пользовательских переменных имеет накопительное свойство. Другими словами, при обработке шаблонов УТ весь пользовательский код на встроенном языке, в известном смысле, образует линейную последовательность, в которой все переменные становятся доступны в любом месте последовательности после их первого определения. Как видно из рисунка, в выражение "after" УТ-1 можно обратиться ко всем переменным, которые были определены в ходе обработки УТ-1.

Синтаксис встроенного языка

Операторы

Последовательность функций и выражений в общем случае выглядит так: 

<оператор>;
<оператор>;
...
<оператор>;
<оператор>


где оператор - функция или выражени.
Как видно из примера, операторы между собой разделяются точкой с запятой.

ВНИМАНИЕ!!! Во многих случаях после последнего оператора блока точка с запятой не ставится.

Под блоком в данном случае понимается место, где вводятся операторы.

Блоком может быть - ветвь then или else условного оператора if; так называемые before и after "выражения" (не путать с выражениями встроеного языка) шаблонов УТ; поле редактирования значения реквизита объекта.

Выражения

Примеры возможных выражений представлены ниже: 

  @__var1 = 23; @__var2 = 40;

// математические выражения
@__var3 = INT(@__var1) + @__var2;
@__var3 = INT(@__var2) * @__var1 + ROUND(@__var3 / 3, 2);

// логические выражения
@__var3 = true;
@__var3 = @__var1 == @__var2; // равно
@__var3 = @__var1 <> @__var2; // не равно
@__var3 = @__var1 >= @__var2; // больше или равно и подобные им: <=, <, >

// соединение строк
@__var1 = "Строка '1'"; @__var2 = 'Строка "2"';
@__var3 = @__var1 + " " + @__var2 // результат: Строка '1' Строка "2"



Отдельно нужно рассмотреть логические выражения и логические операции "AND" и "OR". Во встроенном языке логическая операция "AND" соответствует символу "&", а операция "OR" - символам "||" (две вертикальные черты, следующие друг за другом без пробелов). Логические операции имеют тот же приоритет, что и операции сравнения, поэтому операции сравнения нужно заключать в скобки как показано ниже: 

@__var = (1 == 1) & (2 == 2); // true
@__var = (1 == 2) || (2 == 3); // false



Унарной логической операции отрицания "NOT" найдено не было, но ее можно заменить бинарной операцией сравнения с логическим значением.

Иначе обстоят дела с выражениями, в которых фигурируют даты. В таких выражениях переменные и все выражение в целом нужно явно приводить к типу DATE с помощю одноименной функции DATE, как показано ниже: 

@__var = DATE( DATE(ДАТА()) + 1);
@__var = DATE( DATE("01/02/2006") + 1);

// дата начала текущего месяца относительно опер. дня
@__date = DATE("'01/" + FORMAT( MONTH( ДАТА() ), "99", "INTEGER") + "/" + YEAR( ДАТА() ) + "'")


Условные операторы

Конструкция условного оператора такова: 

  if <условие> then <блок> else <блок> endif


где условие - логическое выражение, блок - последовательность функций и выражений


Ветвь else условного оператора должна быть определена обязательно! Это часто не совсем удобно, но ничего не попишешь :(. Если ветвь else все-таки не должна ничего делать, то можно написать так: 

  if <условие> then <блок> else 1 endif



Одиночные цифры (-1, 0, 1), не являющиеся операндами выражений или параметрами функций, имеют функциональное значение в УТ и предназначены для использования в after и before "выражениях". В данном случае 1 указывает на то, что если условие ложно и должна выполниться ветвь else, то продолжить выполнение операторов, следующих за условием: 1 - продолжить нормальное выполнение.

Если блок операторов ветвей then или else содержит условный оператор, лучше заключить его в круглые скобки так, как показано ниже: 

  if <условие> then ( <блок>; if <условие> then <блок> else <блок> endif; <блок> ) else <блок> endif



В принципе, тот случай, что показан выше не нуждается в скобках, но если убрать блок, стоящий слева или справа от вложенного условия, то скобки нужны обязательно, иначе все выражение не пройдет проверку синтаксиса.

Работа с объектами

Используя тот функционал, который заложен в механизм УТ, с объектами системы можно выполнять любые действия. За исключением такой функции как удаление, все остальные функции реализованы в виде шаблонов действий. Удаление же объектов необходимо выполнять с помощью функции RUN, которая вызывает процедуру Progress 4GL, реализующая поиск и удаление нужных объектов.

С точки зрения объекта, УТ - совокупность шаблонов обработки объектов. Для выполнения действия над одним объектом или некоторым множеством объектов одного класса необходим один шаблон. Другими словами: сколько действий - столько шаблонов. Наиболее часто используемые виды действий: "Создание", "Поиск", "Модификация", "ВизСоздание". Итак, по порядку.

Создание

Создание - это действие которое создает объект класса, указанного в поле "Класс объекта" (или "Класс"). В реквизитах шаблона необходимо определить значения для всех атрибутов объекта, без которых он не может быть сохранен в БД (см. в метасхеме системы параметр "Обязательный" на реквизите класса).

Поиск

Поиск - действие направленное на поиск одного первого объекта определенного класса, удовлетворяющего условиям, наложенным на значения реквизитов. Условия поиска задаются в реквизитах шаблона для каждого индивидуально для каждого реквизита. Порядок поиска или сортировка также задаются в реквизитах шаблона с помощью специализированного реквизита (не из метасхемы), который должен начинаться с символа "$" (доллар).

Чтобы указать в шаблоне, какой реквизит участвует в условии поиска или сортировки, необходимо использовать функцию SEARCH. Согласно описанию, если в условии поиска значение реквизита четко определено, то функция SEARCH вызывается с одним параметром, значение которого равно значению реквизита искомого объекта.

Если в условии поиска значение реквизита заранее не изветно, но известно условие, которому оно должно соответствовать, то функция SEARCH должна вызываться с двумя параметрами, первый из которых равен "?" (знаку вопроса без кавычек), а второй - строковая переменная, определяющая условие. При определении условия поиска, используются выражения, аналогичные тем, которые используются в логических выражениях языка Progress 4GL. Нужно учитывать, что левая часть логического выражения как бы уже определена, а во втрором параметре функции SEARCH необходимо указать лишь правую часть.

При определении условия сортировки, используются выражения "BY" из оператора "FOR" языка Progress 4GL.

Классический пример: нужно найти счет, номер которого начинается с последовательности "42301810". Для этого в выражении реквизита acct вызываем функцию SEARCH(?, "BEGINS('42301810')"). Однако, как нужно задать условие, если необходимо найти объект, значение реквизита которого удовлетворяет некоторому интервалу? Для такого условия выражение будет выглядеть так: 

  op-date = SEARCH(?, "GE " + @open-date(x) + " AND op-date LE " + @end-date(y))



Как видно из примера второе условие явно содержит реквизит. При обработке шаблона парсером УТ для данного реквизита будет построено выражение: 

  WHERE ... (op-date GE '01/01/2006' AND op-date LE '31/12/2006') ...



Когда парсер обрабатывает шаблон поиска объекта, он собирает все выражения реквизитов в одно условие поиска и соединяет их с помощью логической операции AND. Все выражения реквизитов, начинающихся с символа "$", используются парсером для построения условий сортировки. В общем виде выражение, которое создает парсер при обработке шаблона поиска показано ниже: 

  FOR FIRST <имя_таблицы_класса> ...
   WHERE (<выражение_реквизита_1>) [ AND (<выражение_реквизита_2>) ... [ AND (<выражение_реквизита_N>) ]]
   ... 
   <сортировка_BY> 
   ... 
:
...
END.



ВНИМАНИЕ!!! У поиска есть одно ограничение: нельзя искать объект по значению дополнительного реквизита.

Но, теоретически, это ограничение можно обойти, если в последнем выражении шаблона задать условие, которое бы содержало условие выбора записи из таблицы signs. Другими словами, необходимо представить себе, как парсер будет формировать выражение и заставить его включить в него еще одну выборку ;).

Модификация

Модификация использует механизм поиска, но также позволяет задавать значения реквизитов объекта. В реквизитах шаблона сначала должны быть определены условия поиска и сортировки, а следом определения новых значений реквизитов.

ВизСоздание

Когда парсер начинает обрабатывать шаблон действия "ВизСоздание" он первым делом ищет метод form на указанном классе в метасхеме. В случае, если данный метод найден, выполняется процедура Progress 4GL, ему соответствующая. Эти процедуры реализуют вывод экранной формы просмотра и редактирования объекта.

Самые интересные функции

RUN

Функция встроенного языка УТ предназначена для вызова Progress 4GL процедур.

Согласно описанию данной функции, она принимает три параметра: имя процедуры без расширения, список параметров, хранящихся в строковой переменной и разделенных друг от друга запятыми, и флаг возврата значения из процедуры. Важным моментом данной функции является шаблон той 4GL-процедуры, которая должна вызываться. Как показано ниже, шаблон состоит из объявления параметров, блока операций и оператора возврата RETURN. 

/** Блок объявления параметров */

DEF INPUT PARAM iParam1 AS CHAR. DEF INPUT PARAM iParam2 AS DATE. ... 

/** Блок операций */ ... IF some_error THEN RETURN ERROR. ... 

/** Возврат из процедуры */ RETURN.


Если в процедуре не произойдет вызова оператора RETURN, то выполнение УТ завершится с ошибкой. Если оператор RETURN вернет код ERROR, то УТ также завершится с ошибкой и все сделанные ею изменения будут откачены (если в настройках УТ не был изменен реквизит common-op-kind.Transaction на значение "Нет").

Импорт и экспорт данных в текстовом формате

УТ позволяют читать данные из текстового файла и записывать данные в текстовый файл.

Работа с внешними данными

Импорт данных из текстового файла

Импорт данных из текстового файла в УТ реализован не как функция, а как "класс", т.е. для подключения функционала импорта в УТ нужно создать шаблон класса FilterFile. В шаблоне нужно определить значения двух реквизитов данного класса:

  1. fileName - имя импортируемого файла;
  2. codePage - кодировка, в которой сохранен текст (для DOS и UNIX - "ibm866", для Windows - "1251", значения других кодировок можно посмотреть в файле DLC/convmap.cp).

Обрабатывая шаблон фильтра, УТ построчно читает данные из файла, указанного в реквизите fileName. Общее количество прочитанных строк сохраняется в переменной @num-results(n), где n - номер шаблона фильтра.

Для того, чтобы обработать каждую строку, прочитанную из файла, нужно создать еще одну универсальную транзакцию, которую мы будем вызывать из первой с помощью функции ГР_ТРАНЗАКЦИЯ. Вызов функции можно выполнить в выражении "after" шаблона фильтра.

ВНИМАНИЕ! Эту функцию можно вызывать только в "after", т.к., как будет показано ниже, она тесно связана с шаблоном фильтра. Однако, если на шаблоне фильтра реквизит PERSIST установить в значение YES, то при вызове функции ГР_ТРАНЗАКЦИЯ можно сослаться на выборку по значению атрибута FilterTable шаблона фильтра (см. описание функции в системе).

Функиця ГР_ТРАНЗАКЦИЯ вызывает транзакцию, код которой передан ей в качестве параметра, для каждой прочитанной строки. Саму же строку функция сохраняет в переменную @__string, которая доступна в вызываемой транзакции.

Итак, для импорта данных из текстового файла необходимы две УТ:

  1. импортирует данные,
  2. обрабатывает каждую строку.


Проще всего импортировать данные, которые имеют реляционную структуру, но возможен импорт данных с произвольной структурой. Для этого необходимо реализовать немного более сложную логику работы, которую можно показать на одном примере.

Допустим имеется файл, в котором данные сохранены в формате, представленным ниже. 

  <client>
fio=Иванов Иван Иванович
document=66 09 123456 выдан 01.01.2006 тем-то, тем-то
...
</client>

<account>
number=40702810000000000000
open_date=01/01/2006
... 
</account> 

<op>
doc_num=1234
acct_db=20202810000000000000
acct_cr=40702810000000000000
amount=100
... 
</op>

Как видно из примера, в файле есть информация о трех сущностях. Для наглядности предположим, что все объекты из файла необходимо просто создать в базе данных.

Чтобы обработать данный файл необходимо создать две УТ общего назначения и по одной УТ для каждой импортируемой сущьности. Назовем их соответственно: import, parser - УТ общего назначения, client, account, op - транзакции создания объектов. Последовательность вызовов транзакций показана на рисунке ниже.

Транзакция import содержит шаблон класса FilterFile. Для каждой импортируемой строки транзакция import вызывает транзакцию parser с помощью функции ГР_ТРАНЗАКЦИЯ. Выражение "before" шаблона фильтра показано ниже. 

  ...

@__is_tag = false; @__tag_name = ""; 

ГР_ТРАНЗАКЦИЯ("parser") 

<pre>
</code>

Транзакция parser не содержит ни одного шаблона; ее предназначение в том, чтобы разобрать содержимое файла и запустить УТ, 
создающую соотвествующий импортируемый объект. Алгоритм разбора прописан в выражении "after" самой транзакции, текст алгоритма приведен ниже. 

<code>
<pre>
  @__string = TRIM(@__string);

// если это закрывающий тег if (@__string == "</" + @__tag_name + ">") & @__is_tag) then 

  // когда процедура "нашла" закрывающий тег, то предполагается, что все необходимые данные собраны
  // и теперь можно запустить транзакцию, которая создаст определенный объект 
  ТРАНЗАКЦИЯ(@__tag_name);
  @__is_tag = false; @__tag_name = "";

else ( 

  // если это открывающий тег
   if (BEGINS(@__string, "<") & (@__is_tag == false)) then
     @__is_tag = true;
     @__tag_name = TRIM(ENTRY(1, ENTRY(2, @__string, "<"), ">"))
   else (
     // если это выражение
     if (NUM-ENTRIES(@__string, "=") > 1) then (
        @__var = TRIM( ENTRY( 1, @__string, "="));
        @__value = TRIM( ENTRY( 2, @__string, "="));
        // начинаем сохранять данные в переменные, эти переменные будут доступны в транзакциях,
        // вызываемых для создания объектов
        if (@__var == "fio") then @__fio = @__value else 1 endif;
        if (@__var == "document") then @__document = @__value else 1 endif;
        if (@__var == "number") then @__number = @__value else 1 endif;
        if (@__var == "open_date") then @__open_date = DATE(@__value) else 1 endif;
        if (@__var == "doc_num") then @__doc_num = @__value else 1 endif;
        if (@__var == "acct_db") then @__acct_db = @__value else 1 endif;
        if (@__var == "acct_cr") then @__acct_cr = @__value else 1 endif;
        if (@__var == "amount") then @__amount = DEC(@__value) else 1 endif
        // и так далее...

        // хотя возможен и другой вариант: 
        // SETVAR("__" + @__var, @__value)
        // но в этом варианте нельзя управлять типом переменных
     ) else (
       if (@__string == "") then 1 else MESSAGE("Импортируемые данные имеют неправильный формат!", ERROR) endif
     ) endif // выражение
   ) endif // открывающий тег

) endif // закрывающий тег

Все переменные, которые будут созданы в процессе многократного выполнения транзакции parser, будут доступны в транзакциях client, account и op (читайте про область видимости переменных).

В коде предложено два варианта реализации создания списка переменных. Первый - простой, но более гибкий, второй - более универсальный и элегантный, но все переменные, созданные с помощью второго варианта, имеют строковый тип и их приведение необходимо выполнять в момент использования, что не совсем хорошо (если изменится тип переменной, то нужно будет изменить функцию приведения во всех транзакциях, где данная переменная используется).

Импорт данных из XML файла

Импорт из XML возможно организовать более простым способом.

Так же как и при импорте простого текстового файла транзакция будет состоять из двух частей:

  1. Импорт XML во внутренее представление
  2. Обработка с помощью УТ

Для импорта во внтуреннее представление необходимо использовать враппер на ABL (процедура на ABL)

Вот его исходный код. 

/*****************************************
* Файл getXML.p                         *
* Врапер для загрузки XML в УТ		*
*****************************************/
DEFINE INPUT PARAMETER cFileName AS CHARACTER. 
DEFINE NEW SHARED VARIABLE hTTable AS HANDLE.

CREATE TEMP-TABLE hTTable.
hTTable:READ-XML("FILE",cFileName,"EMPTY",?,?,?,?).

RETURN STRING(hTTable).

Вызов враппер располается в блоке "Begin" шаблона УТ, построенной на любом классе. Вот его вызов 

@ResultTable=RUN("getXml","/home2/maslov/myxml.xml","Нет");
ГР_ТРАНЗАКЦИЯ('02',@ResultTable); /* Вызываем УТ, которая будет обрабатывать данные xml файла */

Где /home2/maslov/myxml.xml путь к XML файлу с данными.

На втором шаге необходимо создать УТ которая и будет производить обработку данных XML файла.

Экспорт

Основными инструментами экспорта данных в текстовый файл являются функции встроенного языка УТ WRITE и WRITELN. Обе функции имеют одинаковый синтаксис. Обе принимают два обязательных параметра и два необязательных: строка, имя файла, флаг добавления в конец файла (поумолчанию "Нет") и кодовая страница. Отличие первой функции от второй заключается в том, что вторая добавляет в конец строки символ переноса, а первая нет.
